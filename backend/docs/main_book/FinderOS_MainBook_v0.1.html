1. Ürün Tanımı ve Sınırları

1.1. Ana Ürün

  • FinderOS = nitelikli ürün/servis bazlı lead üretim platformu.
  • Çıktı: doğrulanmış firma/departman kontakları, randevu/appointment akışı, görünürlük (ads) ve ölçüm.

1.2. FinderOS vs FINDIAG

  • FinderOS, FINDIAG’a “supply” sağlar: kategorilere göre firmaları keşfeder, segmentler, iletişim açar, onboarding sağlar.
  • FINDIAG, tüketici tarafında iş atama, teklif alma, token, seçim ve service fee/komisyon kesme akışlarını yürütür.

B1.1. Supply Strategy (DFW başlangıç)

  • Kaynaklar: resmi lisans kayıtları, belediye izinleri, meslek odaları, state contractor registries, Google Business Profiles.

  • FinderOS, sadece FINDIAG’da listelenen kategorileri hedefler.
  • Kategori mapping: external NAICS/keyword → FINDIAG category_code.

B1.3. Geo Targeting: Micro-radius ve “Zero Point”

  • 0.1 mile ≈ 160.9 metre. 20–50 m hedeflemek platform limitlerine bağlıdır.
  • Pratik: pin-drop + geofence (20–50 m) + adres odaklı keyword intent.
  • Risk: çok dar hedefleme reach düşürür; fallback radius policy gerekir.

B1.4. Activation Funnel

  • Verify contact → confirm category & region → SLA prefs → export to FINDIAG.

B1.5. Supply Ops

  • Daily ingestion jobs + enrichment queue + dedupe rules.
  • Human-in-the-loop sadece low confidence contact’larda.

B1.6. Global Scaling

  • Adapter-based source ingestion + unified listing schema.
  • Data moat: vendor graph + response behavior + conversion metrics.

16. Strategic Hardening Backlog

This section summarizes the Master Backlog and establishes non-negotiable positioning and launch readiness requirements.

  • Positioning: SOAR B2B must be positioned as a Market Access / Commercial Intelligence / Precision Exposure OS — not a cheap lead scraper.
  • Partner boundary: The FinderOS ↔ FINDIAG relationship must be documented as a partner / supply-sync architecture with explicit export and idempotency contracts.
  • Primary value surface: The Results Hub is the primary customer value surface and must be visible, reliable, and exportable.
  • Launch gates: Compliance, provenance, observability, and deployment standardization are mandatory before broader launch.

Canonical backlog source: docs/SOARB2B_MASTER_BACKLOG.md.

17. H-019 Controlled Generative UI Foundation

SOAR B2B needs a safe path to present Commercial Intelligence as visual widgets inside Results Hub–class surfaces without opening an unrestricted runtime UI attack surface.

  • H-019 summary: Controlled, template-bound HTML snippets rendered from structured inputs only; embed in iframe sandbox.
  • Why it matters: Positions Results Hub / Intelligence UX as explainable insight, not scraped lists.
  • Approved widget types (foundation): executive_briefing, graph_summary, market_signal_cockpit, opportunity_cluster.
  • Security boundaries: No arbitrary HTML/JavaScript generation, no LLM-driven DOM in parent, no credential exposure, strict allowlist.
  • Deferred: Full unrestricted generative runtime UI / model-executed JS — out of scope until governance and tooling mature.

Operational detail: docs/H-019_GENERATIVE_UI_FOUNDATION.md. DOCX regeneration: see backend/docs/main_book/DOCX_REGEN_NOTE.md when exporting from HTML.

18. H-020 Semantic Capability Layer

Future planner and orchestration workloads need a deterministic, audited description of FinderOS endpoints (purpose, sensitivity, approvals, destructive impact, orchestration allowances) layered on top of OpenAPI payloads.

  • Semantic backend overview: Each curated capability declares risk_level, auth_required, idempotent, orchestration_safe, human_approval_required, sensitive_fields, plus brief schema summaries.
  • Machine-readable manifests: GET /api/v1/system/capabilities returns JSON suitable for tooling without embedding secrets.
  • Deterministic orchestration: Planners ingest explicit metadata—not inferred behavior—preventing hallucinated automation paths.
  • Risk posture: Destructive or customer-impacting operations must advertise higher risk tiers and forbid silent agent execution defaults.
  • Deferred: Autonomous multi-agent choreography, adaptive permissions, runtime registry mutation, unrestricted planner execution.

Operational detail: docs/H-020_SEMANTIC_CAPABILITY_LAYER.md. AI systems must not assume implicit execution authority. manifests describe intent—policy gates, auth flows, budgets, and human approvals still gate real traffic.

DOCX regeneration required from HTML when maintaining Word parity; see backend/docs/main_book/DOCX_REGEN_NOTE.md.

20. H-021 Inference-Aware AI Runtime Optimization Layer

Before attaching heavyweight inference stacks, FinderOS needs deterministic telemetry primitives—token budgets, compaction markers, placeholder routing by tier and task, latency envelopes, and TTFT planning hints—without invoking unmanaged GPU fleets prematurely.

  • AI runtime telemetry: POST /api/v1/system/ai-runtime/profile returns sizing metadata with llm_invoked=false; GET /api/v1/system/ai-runtime/profiles surfaces recent bounded history.
  • Token budgeting: deterministic estimator (len(text)//4) plus safe truncation when compaction is disabled.
  • Prompt compaction: head/tail preservation with explicit marker when context exceeds budgets.
  • Model routing: curated placeholder names (economy-reasoner, standard-reasoner, premium-reasoner) keyed off task + tier.
  • Deferred: Custom GPU serving, TensorRT/vLLM/Triton stacks, speculative decoding, KV-cache engineering—requires measured workloads first.

Operational detail: docs/H-021_AI_RUNTIME_OPTIMIZATION_LAYER.md. Custom GPU serving and KV-cache-level optimization are deferred until measured load justifies them.

DOCX regeneration required from HTML when maintaining Word parity; see backend/docs/main_book/DOCX_REGEN_NOTE.md.

21. H-022 Reflection-Driven Prompt Optimization Layer

Multi-step AI workflows accumulate orchestration and prompt-quality drift. A GEPA-style reflection layer captures structured traces, deterministic revision suggestions, and human-reviewed prompt candidates—without autonomous production mutation.

  • Reflection architecture: Execution traces with scored outcomes, structured failure codes, and append-only optimization history.
  • Deterministic candidates: Template-bound proposals derived from feedback_contracts—no external LLM calls in this foundation.
  • Human approval: Every candidate defaults to pending with human_review_required=true; approve/reject endpoints record explicit identities.
  • Audit trail: Optimization history entries capture approvals/rejections without rewriting production prompts inline.
  • Deferred: Autonomous prompt evolution, RL-style online updates, recursive self-improvement loops, weight fine-tuning.

Operational detail: docs/H-022_REFLECTION_OPTIMIZATION_LAYER.md. Review UI demos: /ui/en/reflection_review_demo.html, /ui/tr/reflection_review_demo.html. Prompt candidates are never automatically promoted into production.

48. H-050 Prompt Cache Governance, Static/Dynamic Context Discipline & Agent Deployment Profile Layer

SOAR B2B treats prompt caching as an architectural discipline: static prefix stability, dynamic suffix isolation, and governed deployment profiles are required for economical long-running AI workflows.

  • Static prefix registry: stable cacheable components with deterministic hashing; rejects volatile metadata (timestamps/random IDs).
  • Dynamic suffix isolation: user turns, tool outputs, runtime observations, and approvals grow after the prefix without mutating it.
  • Cache breakpoints: explicit boundary metadata and explainable invalidation reasons (no hidden invalidation logic).
  • Tool schema + model session stability: detection-only drift signals with cache reset warnings; no mid-session schema mutation.
  • Cache-safe compression: compression instructions are appended as suffix events; the static prefix is never rewritten.
  • Deployment profiles: governed, private-by-default profiles; external channels require identity + firewall + approval.
